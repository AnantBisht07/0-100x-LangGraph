prompts.py
================================================================

Open prompts.py. This is the simplest file in the project but one of the most important by design. Its only job is to store the two system prompts we are testing. Nothing else. No logic, no API calls, no file operations. Just the prompts and one helper function.

Lines 1 to 9 are the docstring. It explains the purpose of the file. In production code, every file starts with a docstring so any developer who opens it immediately understands what it does without reading the code.

Line 11: PROMPT_VERSIONS = {

This opens a dictionary. We write the variable name in all uppercase letters because it is a constant — a value that is set once and never changes while the program is running. Python does not enforce constants the way some languages do, but the uppercase naming is a well-understood convention that signals to other developers: do not modify this value at runtime.

This dictionary is defined at module level, outside any function or class. That means the moment Python imports this file, this dictionary exists in memory. Any other file that writes from prompts import PROMPT_VERSIONS gets access to it immediately.

Line 12: "control": (

This is the first key of the dictionary. The key is the string "control". The parenthesis after the colon opens a grouping — we use this so we can write the string value across multiple lines without Python thinking the statement ended.

Line 13: "You are a concise assistant. Answer clearly and briefly."

This is the actual system prompt for the control version. It gives the model one simple instruction: be concise. We call it the control because it is our baseline — the version we are currently using or the simpler version we want to compare against.

Line 14: ),

The closing parenthesis and comma. The comma separates this key-value pair from the next one.

Lines 15 to 18: "variation": ( "You are a structured, detailed assistant that explains reasoning clearly. " "Break down your answer step by step where appropriate." ),

This is the variation prompt. Notice there are two quoted strings next to each other on lines 16 and 17. When Python sees two string literals placed directly next to each other with no operator between them, it automatically joins them into one single string at compile time. So these two lines become one long string. We split it across two lines purely for readability — the string is too long to fit comfortably on one line.

This prompt is more prescriptive. It tells the model to be detailed, to explain reasoning clearly, and to break things into steps. It is longer than the control prompt because it gives more specific instructions. The whole experiment is designed to measure whether this extra specificity produces better responses according to our metrics.

Line 19: }

Closes the dictionary.

Line 22: def get_prompt(version: str) -> str:

This defines a function called get_prompt. It takes one argument, version, which must be a string — that is what the colon str annotation means. The arrow and str after the closing parenthesis is the return type hint. This function will always return a string. We write these type hints not because Python enforces them at runtime, but because they make the code self-documenting. Any developer reading the function signature instantly knows what goes in and what comes out.

Why write this function at all when you could just write PROMPT_VERSIONS["control"] directly? Two reasons. First, if someone passes a key that does not exist, a direct dictionary access raises a confusing KeyError. This function raises a clear, descriptive ValueError. Second, having a single function as the access point means if we ever change how prompts are stored — say, loading them from a database instead — we only change this function. All callers stay the same.

Line 35: if version not in PROMPT_VERSIONS:

The in keyword checks whether a value exists as a key in the dictionary. Putting not before it inverts the condition. So this line reads: if the version argument that was passed to us is not one of the keys in our dictionary, then run the block below.

Line 36: raise ValueError(

We raise a ValueError. ValueError is Python's built-in exception for when a function receives an argument that has the right type but an invalid value. A string is the right type for version, but "v99" is an invalid value if it is not in the dictionary.

Lines 37 to 38: f"Unknown prompt version '{version}'. " f"Available: {list(PROMPT_VERSIONS.keys())}"

This is the error message. It is an f-string — the f before the quote enables variable interpolation inside curly braces. We embed version in the message so the developer sees exactly what they passed. We call PROMPT_VERSIONS.keys() which returns the dictionary's keys as a view object, then wrap it in list() to convert it into a plain list so it prints cleanly. The developer sees something like: Unknown prompt version 'v99'. Available: ['control', 'variation']. That message is immediately actionable.

Line 40: return PROMPT_VERSIONS[version]

If the version exists in the dictionary, we safely return its value — the corresponding system prompt string. We only reach this line if the if-block above did not raise an exception.

Open evaluator.py. This file measures one LLM response. That is all it does. It receives a text response, a latency measurement, and optionally token data, and it returns a dictionary of numbers that describe that response. It has no knowledge of the API, no knowledge of files, no knowledge of the dashboard. Pure input-output transformation.

Line 11: import random

We import the random module from Python's standard library. We need it on line 49 to generate a simulated user rating.

Line 12: from typing import Optional

Optional is a type annotation helper. Optional[int] means a parameter can be either an integer or the value None. We use this to clearly communicate that total_tokens does not have to be passed by the caller.

Line 15: class Evaluator:

We define a class. A class groups related behaviour together under one name. Here, all metric computation belongs inside Evaluator. Using a class also makes the evaluator easy to replace or extend. The batch runner only knows that it creates an Evaluator and calls .evaluate() on it. If we ever want a different evaluation strategy, we swap out the class.

Lines 16 to 25: This is the class docstring. It lists the four metrics this class computes. Always document what a class does so developers understand its contract without reading every method.

Line 27: def evaluate(

This opens the evaluate method definition.

Line 28: self,

self is the reference to the current instance of the class. Every method in a class receives self as its first parameter automatically. It is how the method can access other attributes or methods of the same object.

Line 29: response_text: str,

The first real parameter. This is the text string that the model returned. The colon str is the type annotation.

Line 30: latency_seconds: float,

The elapsed time in seconds for the API call. This is measured by the batch runner, not here. The evaluator receives it as a ready-made number because measuring time belongs to the code that made the call, not the code that analyses the response.

Line 31: total_tokens: Optional[int] = None,

The total token count from the API response. Optional[int] means it can be an integer or None. The = None after it is the default value — if the caller does not pass this argument at all, it defaults to None. We make it optional because not all API providers return token usage.

Line 32: simulate_rating: bool = True,

A boolean flag that defaults to True. When True, we generate a random rating to stand in for human feedback. When False, no rating is generated. This flag exists so that in the future, when you have real human raters, you can pass False and supply the actual rating yourself.

Line 33: ) -> dict:

Closes the parameter list. The arrow and dict means this method returns a dictionary.

Line 48: word_count = self._count_words(response_text)

We call the private method _count_words, passing in the response text. The result — an integer count of words — is stored in word_count. We delegate this to a separate method so the counting logic is isolated and independently readable. The underscore before _count_words is a Python convention meaning this method is internal. It is not intended to be called from outside the class.

Line 49: user_rating = random.randint(3, 5) if simulate_rating else None

Let us break this down fully. random.randint(3, 5) generates a random integer. The 3 and 5 are both inclusive endpoints, so the result is always one of 3, 4, or 5. We use 3 to 5 rather than 1 to 5 because real user ratings tend to cluster in the upper half of a scale — very few people rate something a 1 or 2 unless it completely failed. The simulated range reflects a realistic distribution.

The if simulate_rating else None part is a ternary expression. It works like this: evaluate the condition simulate_rating, and if it is True, use the left side (random.randint(3, 5)), and if it is False, use the right side (None). So when simulate_rating is True we get a random number. When it is False we get None, and the caller is expected to assign a real rating separately.

Lines 51 to 55: metrics = { "word_count": word_count, "latency_seconds": round(latency_seconds, 4), "user_rating": user_rating, }

We build the output dictionary. Three keys are always present. word_count is the integer from line 48. latency_seconds is the value passed in, rounded to four decimal places with round(). We round here even though the batch runner also rounds, as a defensive measure — the evaluator should always produce consistent precision regardless of what is passed. user_rating is either an integer or None depending on the simulate flag.

Line 58: if total_tokens is not None:

We check whether the caller actually provided a token count. is not None is the correct way in Python to check whether a value is specifically not the None object. We do not write if total_tokens because that would also evaluate as False for 0 — if the token count was zero, we would incorrectly skip adding it.

Line 59: metrics["total_tokens"] = total_tokens

If the token count exists, we add it to the dictionary as a new key. We only add it in this conditional block because if we added it unconditionally, every entry where no token data was available would have "total_tokens": null in the JSON. Null values break the aggregator's averaging logic and make the output file messier. Including the key only when the value is real keeps everything clean.

Line 61: return metrics

We return the finished dictionary. At this point it has either three keys (word_count, latency_seconds, user_rating) or four keys (with total_tokens added) depending on whether token data was available.

Line 63: def _count_words(self, text: str) -> int:

The private word-counting method. Takes a text string, returns an integer count. The return type hint -> int makes this contract explicit.

Line 73: return len(text.split()) if text.strip() else 0

Let us read this from the inside out. text.strip() removes any whitespace characters — spaces, tabs, newlines — from both the beginning and end of the string. In Python, an empty string evaluates as False when used in a boolean context. A non-empty string evaluates as True.

So the ternary reads: if text.strip() is truthy (the string has actual content after stripping), then return len(text.split()). Otherwise return 0.

text.split() with no argument splits the string on any whitespace. It handles single spaces, multiple spaces, tabs, and newlines all the same way, and it discards empty strings that result from consecutive whitespace. So "hello   world" becomes the list ["hello", "world"]. len() counts the elements in that list, giving us 2. This is why we use split() instead of counting individual characters — we want words, not characters.

aggregator.py
================================================================

Open aggregator.py. This file has no imports at all. No external libraries, no standard library, no other modules from our project. It performs pure arithmetic on Python data structures. This is intentional — a file with zero dependencies is always safe to import, easy to test, and impossible to break through a bad library update.

Line 12: def _average(values: list) -> float:

A private helper function. The underscore prefix means it is internal to this module — only the two public functions below should call it. It takes a list and returns a float. We extract this into its own function because we call the averaging logic four times — once per metric — and writing the same code four times would mean if we ever change the logic, we would have to change it in four places.

Line 17: valid = [v for v in values if v is not None]

This is a list comprehension. It builds a new list containing only the elements from values that are not None. We need this filter because some entries in the batch results may have no token data — we stored None in those cases. If we tried to call sum() on a list containing None values, Python would raise a TypeError because you cannot add None to a number. By filtering first, we guarantee valid contains only real numbers.

The syntax reads exactly like English: for every v in values, include v in the new list, but only if v is not None.

Line 18: return round(sum(valid) / len(valid), 4) if valid else 0.0

sum(valid) adds all the numbers in the list. len(valid) gives the count of numbers. Dividing gives the arithmetic mean. round(..., 4) limits the result to four decimal places.

But there is a critical guard here: if valid. After the filter on line 17, valid might be completely empty — if every single entry had None for a given metric. If we tried to divide by len(valid) when valid is empty, len would return 0 and we would get a ZeroDivisionError. The ternary expression prevents this. In Python an empty list is falsy, so if valid evaluates to False when the list is empty. In that case we return 0.0 as a safe default instead of crashing.

Line 21: def aggregate_metrics(results: dict) -> dict:

The main public function. It receives the full results dictionary returned by run_batch_test. That dictionary has "control" and "variation" as keys, each mapping to a list of 20 metric dictionaries. This function returns a new dictionary with the same two keys, but each maps to a single dictionary of averaged values.

Line 47: summary = {}

We initialise an empty dictionary that will be filled during the loop and returned at the end.

Line 49: for version, entries in results.items():

results.items() returns the key-value pairs of the dictionary as an iterable. On the first pass through the loop, version is the string "control" and entries is the list of 20 control metric dictionaries. On the second pass, version is "variation" and entries is the 20 variation dictionaries. We loop generically — we do not hardcode the version names. This means if someone adds a third prompt version to the system, this function handles it automatically with no changes needed here.

Lines 50 to 55: summary[version] = { "average_word_count": _average([e.get("word_count") for e in entries]), "average_latency": _average([e.get("latency_seconds") for e in entries]), "average_tokens": _average([e.get("total_tokens") for e in entries]), "average_user_rating": _average([e.get("user_rating") for e in entries]), }

We assign a new dictionary to summary[version]. Let us walk through one value in detail.

_average([e.get("word_count") for e in entries])

The list comprehension [e.get("word_count") for e in entries] loops through every entry in the entries list. e is one metrics dictionary. e.get("word_count") reads the word_count value from that dictionary. We use .get() rather than direct indexing because .get() returns None if the key is missing, instead of raising a KeyError. The result of the comprehension is a list of 20 values, for example [42, 38, 67, 51, 44, ...]. We pass this list to _average which filters out any None values and computes the mean.

We repeat this pattern four times — once for word_count, once for latency_seconds, once for total_tokens, and once for user_rating. The spaces used to align the right-hand sides of the four dictionary entries are cosmetic — they are just spaces in the source code that make the four lines easier to compare visually.

Line 57: return summary

After both loop iterations complete, summary has two keys — "control" and "variation" — each containing four averaged values. We return it.

Line 60: def detect_regression(summary: dict) -> str | None:

The regression detection function. The return type str | None means it returns either a string message or None. The pipe syntax is Python 3.10 union type notation. The caller checks the return value — if it is None, no regression. If it is a string, regression was detected. We return a string rather than raising an exception or printing directly because the caller decides how to display it. main.py prints it. dashboard.py renders it in a red box.

Line 76: control_rating = summary.get("control", {}).get("average_user_rating")

Two .get() calls chained together. The first, summary.get("control", {}), retrieves the control summary dictionary. The second argument {} is the default — if "control" does not exist in summary for some reason, we get an empty dictionary instead of a KeyError. The second .get("average_user_rating") retrieves the rating value from that dictionary, defaulting to None if the key is missing. So control_rating ends up as either a float or None.

Line 77: variation_rating = summary.get("variation", {}).get("average_user_rating")

Same pattern for the variation. We now have both ratings, each either a number or None.

Lines 79 to 80: if control_rating is None or variation_rating is None: return None

If either rating is missing we cannot make a comparison. We return None — meaning no regression detected. We use the or operator: if the first condition is True we already know the whole expression is True and Python does not even evaluate the second condition. This is called short-circuit evaluation.

Line 82: if variation_rating < control_rating:

This is the regression check. If the variation's average user rating is strictly less than the control's average user rating, the new prompt performed worse. We flag it as a regression.

Line 83: diff = round(control_rating - variation_rating, 4)

We calculate the gap between the two ratings. control_rating minus variation_rating gives a positive number representing how much worse the variation is. Round to four decimal places.

Lines 84 to 88: return ( f"⚠  Potential regression detected. " f"Variation rating ({variation_rating}) is {diff} points " f"below control ({control_rating})." )

We build and return the warning string. Three f-strings next to each other inside parentheses get joined into one string. We include the exact numbers — variation_rating, diff, and control_rating — so the developer immediately sees the magnitude of the regression without having to look anything up. The ⚠ symbol is a Unicode warning character that draws attention in the terminal.

Line 90: return None

If we reached this line, variation_rating was not less than control_rating — it was equal or greater. The variation is performing at least as well as the control. No regression. We return None.
